1. Descripción de la Actividad

El objetivo de esta práctica es evolucionar la aplicación web desarrollada previamente, dotándola de una interfaz de programación de aplicaciones (API JSON). Esta API servirá como puente de comunicación para una futura aplicación móvil.

Una vez implementada, el foco principal será la seguridad. Debereis actuar como auditores de vuestro propio código, siguiendo los estándares internacionales de OWASP.

2. Requerimientos Técnicos

2.1. Desarrollo

Implementar endpoints para las operaciones CRUD principales de la aplicación.

  • Ver datos registrados
  • Registrar nuevos datos

2.1.1. Formato de datos

JSON.

2.1.2. Autenticación

Uso de tokens (ej. JWT) u OAuth2.

2.1.3. Documentación

Para que la aplicación móvil pueda integrarse correctamente, se puede documentar la API siguiendo el estándar OpenAPI 3.0.

Pasos a seguir:

  1. Instalación de Dependencias: Dependiendo del framework (ej. springdoc-openapi para Spring Boot, swagger-jsdoc para Node.js/Express, o drf-spectacular para Django), instalar la librería necesaria.
  2. Definición de Schemas: Cada modelo de datos (User, Product, etc.) debe tener su esquema definido para que el cliente móvil conozca los tipos de datos (string, integer, boolean).
  3. Anotación de Endpoints:
  4. Summary/Description: Explicar qué hace el endpoint.
  5. Responses: Definir los códigos de estado HTTP (200 OK, 201 Created, 401 Unauthorized, 403 Forbidden, 500 Error).
  6. Security: Marcar qué rutas requieren el token JWT/OAuth2.

Ejemplo de estructura de un Endpoint en Swagger: 

/auth/login:
post:
summary: Autenticación de usuario
responses:
200:
description: Token generado con éxito
401:
description: Credenciales inválidas

3. Fase de Auditoría de Seguridad (OWASP)

Los alumnos deberán evaluar la seguridad de su API basándose en la OWASP Web Security Testing Guide (WSTG) y el API Security Top 10.

Puntos clave a testear:

  • WSTG-IDNT-04: Prueba de robustez de la autenticación.
  • WSTG-AUTH-02: Prueba de controles de autorización (IDOR).
  • WSTG-CRYP-03: Seguridad en el transporte (Uso de TLS/HTTPS).

4. Guía de Referencia y Recursos

Para realizar las pruebas, se deben consultar los siguientes documentos oficiales:

  • OWASP Web Security Testing Guide (WSTG) v4.2
  • OWASP API Security Top 10

5. Entregables y Documentación

Los alumnos deben entregar enlace al repositorio o un archivo comprimido:

5.1. El Código Fuente

  • Implementación de la API.
  • Archivo de configuración del entorno.

5.2. Documento de Pruebas (Security Report)

Debe seguir el siguiente formato de tabla para cada vulnerabilidad probada:

ID Control (WSTG) Descripción de la Prueba Resultado (Pass/Fail) Observaciones/Mitigación
WSTG-CONF-05 Revisar cabeceras HTTP Pass Se incluye HSTS
WSTG-AUTH-03 Bypass de sesión Fail Token expira en 24h

5.3. Interfaz API

Se debe incluir una captura de pantalla de la interfaz Swagger UI funcionando y el archivo de definición en formato JSON o YAML (openapi.json).

Se evaluará que:

  • Todos los endpoints aparezcan en la interfaz.
  • Se hayan definido los modelos de error (4xx y 5xx).
  • Se pueda probar la API directamente desde el "Try it out" de Swagger, incluyendo el paso de autenticación.

6. Criterios de Evaluación

  • Funcionalidad (40%): La API responde correctamente a las peticiones del cliente móvil.
  • Seguridad (40%): Calidad de la auditoría y corrección de vulnerabilidades detectadas.
  • Documentación (20%): Claridad en el reporte de pruebas y uso correcto del formato Markdown si es el caso.

7. Notas

Resumen rápido de documentación de API REST según la tecnología:

  • Node.js (Express): Suelen usar swagger-ui-express y swagger-jsdoc. Escriben comentarios JSDoc encima de sus rutas y la librería genera el JSON.
  • Java (Spring Boot): Basta con añadir la dependencia de springdoc-openapi-ui. Al arrancar la app, la documentación se genera automáticamente en /swagger-ui.html analizando los @RestController.
  • Python (FastAPI): Es nativo. FastAPI genera la documentación en /docs automáticamente mientras definen las funciones.
  • PHP (Laravel): Utilizan el paquete L5-Swagger (basado en swagger-php).
