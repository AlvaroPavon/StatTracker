1. Objetivo

El alumno deberá fortalecer una aplicación Android integrando controles de la categoría MSTG-RESILIENCE de OWASP. El objetivo es implementar mecanismos de defensa activa que dificulten el análisis dinámico y la manipulación (tampering) de la aplicación en tiempo de ejecución.

2. Requisitos técnicos: Niveles de resiliencia (MSTG-RESILIENCE)

Para cada nivel, se debe implementar la lógica correspondiente y documentar su efectividad.

2.1. Nivel 1: Detección de root y entorno inseguro (MSTG-RES-1)

Control
La aplicación debe detectar si se ejecuta en un dispositivo con privilegios de root.
Implementación
  • Por ejemplo, comprobar existencia de binarios en `/system/xbin/su`.
  • Integrar la librería RootBeer para una detección multivariable.
Política
Si se detecta root, la app debe informar al usuario y finalizar la ejecución de forma segura.

2.2. Nivel 2: Anti-Debugging y Prevención de Análisis Dinámico (MSTG-RES-2)

Control
La aplicación debe evitar ser depurada por herramientas externas.
Implementación
  • Configurar android:debuggable="false" en el manifiesto de release.

  • Implementar chequeo dinámico en el `onCreate` de la actividad principal: 

    if (android.os.Debug.isDebuggerConnected()) { System.exit(0); }

2.3. Nivel 3: Verificación de integridad y firma (MSTG-RES-3)

Control
La aplicación debe detectar si su código o recursos han sido modificados (Anti-Tampering).
Implementación
  • Implementar un método que verifique el Hash del certificado de firma de la APK en tiempo de ejecución.
  • Comparar con el valor hardcodeado original para detectar re-empaquetado.

2.4. Nivel 4: Ofuscación de código y empaquetado (MSTG-RES-4)

Control
El código debe ser difícil de leer tras la ingeniería inversa.
Implementación
  • Habilitar minifyEnabled true en build.gradle.
  • Configurar proguard-rules.pro para proteger nombres de clases y métodos críticos sin romper dependencias.

2.5. Nivel 5: Respuesta dinámica y resiliencia UX

Control
La aplicación debe reaccionar de forma proactiva a las amenazas detectadas.
Implementación
  • Asegurar un cierre controlado. No debe haber crashes técnicos, sino mensajes claros de seguridad al usuario.
  • (Opcional) Implementar detección de emuladores para evitar el análisis en entornos controlados.

3. Evaluación

Criterio Excelente (10) Adecuado (5) Insuficiente (0)
MSTG-RES-1/2 (40%) Bloqueo efectivo de Root/Debug con gestión de salida. Detección básica fácil de saltar. Sin protección.
MSTG-RES-3/4 (40%) Código ofuscado e integridad verificada (Firma). Solo ofuscación básica o errores de firma. Sin protección estática.
Resiliencia/UX (20%) Cierre controlado y mensajes claros de seguridad. Cierre brusco (crash) sin mensaje. Permite acceso inseguro.

4. Entrega

  • Archivo APK resultante y código fuente (`.zip` o enlace a Repo).
  • Informe README.md o .pdf con evidencias (capturas de pantalla) de las protecciones funcionando en entornos de prueba (ej. ADB o Genymotion con root).
